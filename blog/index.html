The Routing Decision Nobody Talks About #

Every enterprise deploying LLMs makes a routing decision -- whether they realize it or not. A basic load balancer in front of vLLM replicas is a routing decision. A managed API gateway is a routing decision. Each one carries trade-offs in cost, latency, vendor lock-in, and operational flexibility that compound at scale.

This post maps the inference routing landscape: what approaches exist, where each one wins, and how to choose based on your workload.

The Routing Approaches #

Round-robin / random

The default. Requests are distributed evenly across replicas with no awareness of what each replica has cached or how busy it is.

  • Strengths: Zero configuration, works everywhere, no single point of failure
  • Weaknesses: Ignores KV cache state entirely. Every request is a cold prefill. GPU waste scales linearly with traffic.
  • Best for: Single-model deployments with uniform, short prompts

Load-aware routing

Routes to the least-busy replica based on queue depth, GPU utilization, or pending request count.

  • Strengths: Prevents hotspots, simple to implement
  • Weaknesses: Still ignores cache state. A lightly-loaded pod with no cached prefix is worse than a busy pod with a warm cache.
  • Best for: Heterogeneous hardware where some replicas are faster than others

Hash-based routing

Hashes the prompt (or a prefix of it) and consistently maps it to the same replica.

  • Strengths: Deterministic cache affinity, no state tracking needed
  • Weaknesses: Rigid. Doesn't handle variable-length prefixes, pod scaling, or cache eviction. Skewed hash distributions create hotspots.
  • Best for: Static workloads with predictable prompt patterns

Prefix-cache-aware routing (llm-d)

Tracks actual KV cache state across all replicas in real time. Routes each request to the pod with the longest matching cached prefix, balanced against load.

  • Strengths: Maximizes cache hits, adapts to dynamic workloads, handles variable prefix lengths, scales with pod count
  • Weaknesses: Requires EPP sidecar, adds a routing hop, more complex to deploy
  • Best for: Multi-turn conversations, agentic workflows, document Q&A, any workload with prefix sharing
Key Takeaway

The gap between routing approaches widens dramatically as prefix length grows. At 32K tokens, cache-aware routing is 47.5x faster than round-robin -- the difference between 3.5 seconds and 84 milliseconds.

Vendor lock-in: the hidden cost

Proprietary inference platforms often bundle routing with their serving stack. You get convenience today, but when you need to switch hardware vendors, change models, or move to a different cloud, you're rebuilding the routing layer from scratch.

llm-d runs on any Kubernetes cluster, any GPU vendor (NVIDIA, AMD, Intel), and any vLLM-supported model. The routing logic is decoupled from the infrastructure.

Hardware flexibility

As GPU generations evolve and new accelerators emerge (NVIDIA B200, AMD MI325X, Intel Gaudi 3), your routing layer shouldn't be tied to one vendor's SDK. llm-d uses standard Kubernetes primitives -- pods, services, endpoints -- that work across any hardware.

The Business View #

For decision-makers who need the summary:

The question isn't "which routing is fastest?" It's "which routing approach lets us scale without locking us in, works across GPU vendors, and has a community behind it?" Speed is table stakes -- llm-d delivers 47.5x TTFT improvement. The strategic advantage is what comes with it.
Key Takeaway

Vendor lock-in is the hidden cost of proprietary inference platforms. llm-d runs on any Kubernetes cluster, any GPU vendor, and any vLLM-supported model -- keeping your options open as hardware evolves.

How to Choose #

Use this decision tree:

  1. Are your prompts short and uniform? Round-robin is fine. Don't over-engineer.
  2. Do you have prefix sharing? (Multi-turn, agents, document Q&A) --> You need cache-aware routing.
  3. Are you on multiple GPU vendors? --> Avoid vendor-specific routing. Use Kubernetes-native.
  4. Do you need multi-tenant isolation? --> llm-d supports per-namespace routing policies on OpenShift.
