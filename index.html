Chapter Four

The Verdict

How we characterized each deployment stack, where the data comes from, and an honest take on when cache-aware routing is worth it.

When Cache-Aware Routing Is Worth It -- and When It Isn't

Worth the investment when...
  • Your workload has high prefix overlap -- multi-turn chat, agentic workflows, RAG with shared templates
  • You are already running Kubernetes with KServe or planning to
  • Context lengths regularly exceed 4K tokens, where cache savings compound
  • You need consistent low-latency TTFT, not just good average throughput
  • You are scaling to multiple GPU pods and cache misses are wasting expensive compute
Probably not worth it when...
  • Your prompts are mostly unique with little prefix reuse -- batch processing, one-shot classification
  • You are running a single GPU pod -- local caching already handles everything
  • You do not use Kubernetes and do not plan to -- the Platform stack adds real complexity
  • Context lengths are short (under 1K tokens) -- the prefill savings are minimal
  • Simplicity matters more than performance -- the Basic stack is genuinely simpler to operate
Benchmark Hardware & Configuration

TTFT benchmarks were run on NVIDIA H200 GPUs with Llama 3.1 70B using vLLM as the runtime across all three stacks. Context lengths range from 500 to 32,768 tokens. The KV cache hit rate models 80--95% prefix reuse for multi-turn and RAG workloads, reflecting production patterns with shared system prompts.

Stack-Based Comparison

This comparison evaluates three deployment stacks, not individual products. Every stack uses the same runtime (vLLM or SGLang) -- the difference is the routing and deployment layer. The Basic stack uses simple round-robin. The Platform stack adds KServe for Kubernetes-native operations. The Cache-Aware stack adds llm-d's EPP for KV-cache-aware routing.

Cache Savings Estimates

Cache-aware savings come from routing requests to pods with warm KV caches. At 32K context with high prefix overlap, llm-d's EPP achieves up to 47.5x TTFT reduction vs round-robin. Savings vary by workload: multi-turn chat sees ~40x, RAG ~35x, agentic ~30x. Batch processing with unique prompts shows minimal cache benefit -- round-robin may actually be preferable.

Data Sources & Limitations

Data sources include llm-d project benchmarks, vLLM/SGLang documentation, KServe architecture docs, and CNCF project materials. Limitations: local prefix caching within a pod (provided by the runtime) is present in all three stacks -- the cache-aware advantage comes from cross-pod coordination. Real-world results depend on hardware, model size, prompt patterns, and cluster configuration.
